За два дні — 1 квітня 2026 — [KSeF стає обов'язковим](/uk/blog/ksef-od-1-kwietnia-2026) для всіх активних платників ПДВ. Якщо ти маєш клієнтів за кордоном — обов'язок стосується тебе так само. Рахунок для контрагента з Берліна, Амстердама чи Нью-Йорка має потрапити до KSeF — навіть якщо цей контрагент не має польського NIP і ніколи не увійде до системи.

Іноземний покупець не має доступу до KSeF. Тобі потрібно надіслати йому читабельну копію рахунку окремо — на пошту, мейлом, як і раніше. Цей матеріал покаже крок за кроком, як правильно заповнити дані іноземного контрагента та обрати ставку ПДВ.

---

## Покупець з ЄС із номером VAT-UE

Найтиповіший випадок для ФОП-фрилансерів: ти виставляєш рахунок компанії з Євросоюзу, яка має активний номер VAT-UE.

У даних покупця (секція `Podmiot2` у FA(3)) використовуй пару полів `KodUE` + `NrVatUE` — **не NIP**. Поле NIP призначене виключно для польських податкових номерів. Іноземний номер VAT, вписаний у NIP, призведе до відхилення рахунку системою KSeF.

### Які поля заповнити

- **`KodUE`** — двобуквений код країни ЄС (наприклад, `DE` — Німеччина, `NL` — Нідерланди, `FR` — Франція)
- **`NrVatUE`** — номер VAT покупця **без префікса країни** (сам номер, без «DE» на початку)
- **`Nazwa`** — повна назва компанії покупця

В адресі покупця вказуєш `KodKraju` (той самий код, що й у `KodUE`), `AdresL1` (вулиця та номер) і `AdresL2` (поштовий індекс та місто).

<Tip>
  Перед виставленням рахунку перевір номер VAT-UE контрагента в системі VIES — офіційній базі
  Європейської комісії. Неактивний номер означає проблеми з розрахунком WDT (wewnątrzwspólnotowa
  dostawa towarów — внутрішньоспільнотна поставка товарів) або послуг.
</Tip>

Приклад — рахунок для компанії з Німеччини:

<XmlExample copyable>
  ```xml
  <Podmiot2>
    <DaneIdentyfikacyjne>
      <KodUE>DE</KodUE>
      <NrVatUE>123456789</NrVatUE>
      <Nazwa>Berlin Design GmbH</Nazwa>
    </DaneIdentyfikacyjne>
    <Adres>
      <KodKraju>DE</KodKraju>
      <AdresL1>Musterstraße 1</AdresL1>
      <AdresL2>10115 Berlin</AdresL2>
    </Adres>
    <JST>2</JST>
    <GV>2</GV>
  </Podmiot2>
  ```
</XmlExample>

`JST` і `GV` — обов'язкові поля в кожному блоці `Podmiot2` структури FA(3). Для іноземних контрагентів завжди вказуй значення `2` («не стосується»). Без цих полів рахунок не пройде валідацію.

## Покупець з ЄС без номера VAT

Не кожен контрагент із ЄС має номер VAT-UE — приватна особа, фрилансер без реєстрації, фірма без VAT. У такому випадку замість `KodUE` + `NrVatUE` використовуй поле `BrakID` зі значенням `1` («покупець не має ідентифікаційного номера»).

<Warning>
  Без номера VAT-UE контрагента ти не можеш застосувати ставку `0 WDT` на товари або `np II` на
  послуги. Правила оподаткування можуть бути іншими — проконсультуйся з бухгалтером.
</Warning>

## Покупець не з ЄС

Клієнт зі США, Великої Британії (після Брекзіту), Швейцарії, Канади — це інша категорія. Тут не використовуєш `KodUE` і `NrVatUE` — ці поля лише для покупців з Євросоюзу.

Якщо покупець має податковий ідентифікатор у своїй країні — вказуєш пару `KodKraju` + `NrID` у блоці `DaneIdentyfikacyjne`:

- **`KodKraju`** — двобуквений код ISO країни (наприклад, `US`, `GB`, `CH`, `NO`)
- **`NrID`** — іноземний податковий номер (наприклад, EIN у США, UTR у Великій Британії)

Приклад — рахунок для компанії зі США:

<XmlExample copyable>
  ```xml
  <Podmiot2>
    <DaneIdentyfikacyjne>
      <KodKraju>US</KodKraju>
      <NrID>12-3456789</NrID>
      <Nazwa>US Design Inc.</Nazwa>
    </DaneIdentyfikacyjne>
    <Adres>
      <KodKraju>US</KodKraju>
      <AdresL1>123 Main Street</AdresL1>
      <AdresL2>New York, NY 10001</AdresL2>
    </Adres>
    <JST>2</JST>
    <GV>2</GV>
  </Podmiot2>
  ```
</XmlExample>

Зверни увагу: `KodKraju` зустрічається двічі — у `DaneIdentyfikacyjne` (контекст податкової ідентифікації) і в `Adres` (адреса). Це два різних поля в різних секціях — обидва обов'язкові.

Якщо покупець не має ідентифікатора — використовуй `BrakID` зі значенням `1`, так само як для покупця з ЄС без номера.

<Info>
  Велика Британія після Брекзіту не є країною ЄС. Контрагент із UK — це покупець не з ЄС.
  Використовуй `KodKraju` = `GB` і `NrID`, а не `KodUE` + `NrVatUE`.
</Info>

## Ставки ПДВ на рахунках для іноземних контрагентів

Це найлегше місце для помилки. Ставка ПДВ на рахунку залежить від того, **що** ти продаєш (товар чи послугу) і **кому** (покупець з ЄС чи не з ЄС).

| Тип операції             | Ставка (P_12) | Поле суми нетто | Коли застосовувати                                                                      |
| :----------------------- | :------------ | :-------------- | :-------------------------------------------------------------------------------------- |
| WDT — товари до ЄС       | `0 WDT`       | `P_13_6_2`      | Продаж товарів покупцю з номером VAT-UE                                                 |
| Експорт товарів          | `0 EX`        | `P_13_6_3`      | Продаж товарів покупцю не з ЄС                                                          |
| Послуги для фірм з ЄС    | `np II`       | `P_13_9`        | Послуги згідно з ст. 100 ч. 1 п. 4 Закону про ПДВ — покупець сплачує ПДВ у своїй країні |
| Послуги для фірм не з ЄС | `np I`        | `P_13_8`        | Послуги поза територією Польщі (ст. 28b) — покупець сплачує ПДВ                         |

### Як це виглядає на практиці?

Робиш вебсайт для клієнта з Німеччини з номером VAT-UE? → Це послуга, покупець сплачує ПДВ у себе, ставка **`np II`**.

Робиш вебсайт для клієнта зі США? → Це послуга поза територією ЄС, ставка **`np I`**.

Відправляєш фізичний товар клієнту з Нідерландів (із номером VAT-UE)? → Це WDT, ставка **`0 WDT`**.

Відправляєш фізичний товар клієнту з Великої Британії? → Це експорт товарів, ставка **`0 EX`**.

<Warning>
  Ставка `oo` (odwrotne obciążenie — вітчизняне зворотне нарахування) **ніколи** не стосується
  іноземних контрагентів. Для покупців з ЄС і не з ЄС використовуй `np I` або `np II`. Використання
  `oo` для іноземного покупця — одна з найчастіших помилок і причина відхилення рахунку.
</Warning>

Значення ставок у FA(3) **чутливі до регістру та пробілів**. Правильно: `np I`, `np II`, `0 WDT`, `0 EX`. Неправильно: `NP I`, `npI`, `0WDT`.

## TL;DR

- Рахунок для іноземного контрагента **має** потрапити до KSeF — так само як і вітчизняний.
- Іноземний покупець не має доступу до KSeF — надішли йому читабельну копію окремо (PDF із QR-кодом).
- Покупець з ЄС із номером VAT-UE → `KodUE` + `NrVatUE` (не NIP).
- Покупець з ЄС без номера VAT → `BrakID` = `1`.
- Покупець не з ЄС з ідентифікатором → `KodKraju` + `NrID`.
- Покупець не з ЄС без ідентифікатора → `BrakID` = `1`.
- Послуги для фірм з ЄС → ставка `np II`.
- Послуги для фірм не з ЄС → ставка `np I`.
- WDT (товари до ЄС) → ставка `0 WDT`.
- Експорт товарів → ставка `0 EX`.
- **Ніколи** не використовуй ставку `oo` для іноземних контрагентів.
- Перевір номер VAT-UE в [VIES](https://ec.europa.eu/taxation_customs/vies/) перед виставленням рахунку.

<Tip>
  Завантаж свій XML-файл у валідатор на [ksefuj.to](/) перед надсиланням до KSeF — безплатно
  перевіриш, чи дані іноземного покупця заповнені правильно. Твій файл не виходить з браузера.
</Tip>
